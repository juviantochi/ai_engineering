background:
- Enrichment flow executes at the end of merchant onboarding. BNPL sends Kafka events which trigger fs-brick; fs-brick generates enrichment data and calls back BNPL with the processing status and an S3 path containing the payload.
- `merchant_enrichment_data` captures each enrichment attempt (batchId + enrichment type) along with merchant linkage and the S3 path of the delivered payload.
- We assume the S3 payloads match the Veefin borrower-data-inquiry responses (see `documentation/request/veefin-data-point/sample_FDC.json` and `sample_PEFINDO.json`) until real samples arrive.

clarified requirements:
- BNPL must persist scalar credit summary fields (no JSONB) from the S3 payload into a new table `merchant_credit_summary`. The columns and mapping rules are:

| Column                        | Type                     | Source                                                                    | Notes                                        |
|-------------------------------|--------------------------|---------------------------------------------------------------------------|----------------------------------------------|
| `merchant_credit_summary.id`  | UUID                     | generated by BNPL                                                         | Primary key.                                 |
| `merchant_id`                 | UUID                     | `merchant_enrichment_data.merchant_id`                                    | Foreign key to `merchant_data`.              |
| `pefindo_batch_id`            | VARCHAR(60)              | `merchant_enrichment_data.batch_id`                                       | Stored for traceability.                     |
| `fdc_batch_id`                | VARCHAR(15)              | derived (`FDC`/`PEFINDO`)                                                 | Mirrors enrichment type.                     |
| `status`                      | VARCHAR(32)              | Enrichment data status `READY`/`NOT_READY`                                | only valid if status is `READY`. only `READY` |
| `total_fasilitas`             | INT                      | FDC count of `pinjaman`; PEFINDO `jml_aktif_fasilitas + jml_tutup_fasilitas` | Veefin `jmlFasilitas`.                       |
| `jml_hari_tunggakan_terburuk` | INT                      | PEFINDO `jml_hari_tunggakan_terburuk`                                     | Veefin `jmlHariTunggakanTerburuk`.           |
| `jml_saldo_terutang`          | NUMERIC(20,2)            | PEFINDO `jml_saldo_terutang`             | Veefin `jmlSaldoTerutang`.                   |

additional clarifications:
- Relation is enforced via `merchant_enrichment_data.pefindo_batch_id` + `merchant_id`. Each enrichment callback inserts one summary row; no updates expected (insert-once semantics).
- The S3 path itself stays in `merchant_enrichment_data`; no duplication in the new table.
- BNPL will expose this stored summary to Veefin through an API endpoint (path TBD during implementation) once data persistence is in place.
